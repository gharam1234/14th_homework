핵심요구사항) playwright 테스트를 활용하여 TDD 기반으로 구현하고, 테스트에 통과할 때까지 반복할 것.

1) 테스트 제외 라이브러리
   - jest
   - @testing-library/react

2) 테스트 조건
   - timeout은 설정하지 않거나, 500ms 미만으로 설정할 것.
   - 페이지가 완전히 로드된 후 테스트할 것.
       - 페이지 로드 식별 요구사항: 고정식별자 data-testid 대기 방법
       - **중요금지사항** 페이지 로드 식별 금지사항: networkidle 대기 방법

3) 테스트 API 조건
3-1) 데이터
    - 실제 Supabase 데이터를 사용할 것.
    - Mock 데이터를 사용하지 말 것.

3-2) 성공 시나리오
    - Supabase에서 해당 전화기 ID의 데이터 조회 성공
    - 바인딩된 필드들이 UI에 올바르게 표시됨

3-3) 실패 시나리오
    - 데이터 조회 실패 또는 빈 결과 처리
    - 에러 메시지 표시

4) 데이터 조건
    - 저장소: Supabase (phones 테이블)
    - 요청방식: 실시간 구독 또는 단일 조회 (useEffect + supabase client)
    - 요청 파라미터 구조:
      phones 테이블에서 특정 ID의 레코드 조회
      SELECT * FROM phones WHERE id = [phone_id]

핵심요구사항) 현재 DUMMY_PHONE_DATA를 제거하고, 실제 Supabase 데이터를 바인딩할 것.
                - 모든 하드코딩된 Mock 데이터 제거
                - 실제 phones 테이블 데이터만 사용

핵심요구사항) 다음 필드들을 Supabase phones 테이블에서 바인딩할 것.
                1) 상세 설명 섹션 (프롬프트 1)
                    - description: 상세 설명 텍스트 (필수)
                        데이터 타입: text
                        렌더링: 포맷 그대로 유지 (줄바꿈/들여쓰기 유지)

                2) 거래 희망 지역 섹션 (프롬프트 2)
                    - latitude: 위도 (선택)
                        데이터 타입: numeric
                        유효 범위: -90 ~ 90

                    - longitude: 경도 (선택)
                        데이터 타입: numeric
                        유효 범위: -180 ~ 180

                    - address: 거래 주소 (선택)
                        데이터 타입: varchar

                    - address_detail: 상세 주소 (선택)
                        데이터 타입: varchar

                3) 기타 바인딩 필드
                    - title: 상품명 (필수)
                    - summary: 한줄 요약 (선택)
                    - price: 판매 가격 (필수)
                    - device_condition: 기기 상태 (선택)

핵심요구사항) Hook 함수를 다음과 같이 구성할 것.
                1) Hook 이름: useFetchDetail
                2) 입력 파라미터
                    - phoneId: string (필수, URL 파라미터에서 추출)

                3) 반환값
                    - data: PhoneDetailType | null (조회된 데이터)
                    - loading: boolean (데이터 로딩 중 여부)
                    - error: Error | null (에러 객체)
                    - refetch: () => void (데이터 새로고침 함수)

                4) Hook 로직
                    - useEffect에서 Supabase 실시간 구독 또는 단일 조회 설정
                    - 컴포넌트 언마운트 시 구독 해제
                    - 에러 처리: console.error로 에러 로깅

핵심요구사항) 컴포넌트에서 Hook을 사용하여 조건부 렌더링을 구현할 것.
                1) 로딩 상태 (loading === true)
                    - 로딩 스피너 또는 "로딩 중..." 메시지 표시
                    - data-testid="phone-detail-loading"

                2) 성공 상태 (data !== null && !loading)
                    - 바인딩된 필드들 렌더링
                    - 각 섹션: 데이터가 없으면 "등록된 정보가 없습니다" 표시

                3) 에러 상태 (error !== null)
                    - 에러 메시지 표시
                    - data-testid="phone-detail-error"
                    - 재시도 버튼 제공 (refetch 호출)

핵심요구사항) 데이터 바인딩 상세내용을 다음과 같이 구현할 것.
                1) 상세 설명 필드
                    - 소스: data.description
                    - 렌더링: <pre> 또는 <div style={{whiteSpace: 'pre-wrap'}}>
                    - 조건부: data.description ? <...> : <"등록된 정보가 없습니다...">

                2) 거래 희망 지역 필드
                    - 소스: data.latitude, data.longitude, data.address
                    - 렌더링: 좌표 유효성 검사 후 지도 표시
                    - 조건부: (latitude && longitude) ? <KakaoMap> : <"등록된 정보가 없습니다...">

                3) 기타 필드 (상단에 이미 있다면 확인만)
                    - title, summary, price, device_condition 등
                    - 기존 바인딩 유지

핵심요구사항) CSS 처리를 다음과 같이 구현할 것.
                1) 공백/줄바꿈 유지
                    - description 필드: white-space: pre-wrap 또는 pre-line
                    - 불릿, 들여쓰기 등 모든 포맷 유지

                2) 색상 및 타이포그래피
                    - global.css의 토큰 활용 (하드코딩 금지)
                    - 섹션 컨테이너: 일관된 배경색/패딩
                    - 텍스트: 가독성 좋은 크기/색상

핵심요구사항) Playwright 테스트를 작성할 것.
                1) 테스트 케이스
                    1-1) 데이터 로드 테스트
                        - 페이지 로드 시 data-testid="phone-detail-loading" 표시 확인
                        - 데이터 로드 완료 후 실제 데이터 표시 확인

                    1-2) 필드별 바인딩 테스트
                        - description이 올바르게 표시됨
                        - latitude/longitude가 유효할 때 지도 표시됨
                        - 유효하지 않을 때 "등록된 정보가 없습니다" 표시됨

                    1-3) 에러 처리 테스트
                        - 조회 실패 시 에러 메시지 표시
                        - 재시도 버튼 클릭 시 데이터 새로고침

                2) 테스트 작성 방식
                    - 실제 Supabase 데이터 사용 (개발 환경 기준)
                    - URL 파라미터로 유효한 phone_id 전달
                    - 로딩 완료 대기: page.locator('[data-testid="phone-detail-description"]').waitFor()

핵심요구사항) 성능 최적화를 고려하여 구현할 것.
                1) 불필요한 리렌더링 방지
                    - useMemo, useCallback 활용
                    - 파생 상태는 최소화

                2) 구독 관리
                    - Supabase 구독은 정확히 설정 해제 (메모리 누수 방지)
                    - 컴포넌트 언마운트 시: unsubscribe() 호출

핵심요구사항) 다음 조건을 적용할 것.
                1) 에러 로깅
                    - console.error 사용 (프로덕션에서는 로깅 서비스 활용)

                2) 타입 안정성
                    - TypeScript 타입 정의: PhoneDetailType, PhoneDetailData 등
                    - 선택적 필드는 ? 명시

                3) 재사용성
                    - Hook은 다른 컴포넌트에서도 사용 가능하도록 설계
                    - 로직과 UI 분리
