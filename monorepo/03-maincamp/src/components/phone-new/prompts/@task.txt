태스크2. 입력 검증 및 에러 처리 모듈@

요구사항 1. 필수로 꼭 입력해야 하는지, 가격이 올바른 형식인지, 좌표 값이 허용된 범위 안에 있는지, 태그가 서로 중복되지 않는지, 글자 수 제한을 넘지 않았는지 등을 검사하는 함수를 따로 모아서 “유효성 검사 모듈”로 분리한다. 이 모듈은 동기 방식과 비동기 방식 검사를 모두 지원해야 한다.

요구사항 2. 사용자가 폼을 제출하면, 모든 입력 필드를 한 번에(병렬로) 검사한다. 검사 결과, 문제가 있는 필드들의 목록과 각 필드에 대한 에러 메시지를 호출한 쪽에 넘겨준다. 모든 필드가 검사에 통과하면, 공백 제거·숫자/문자 형 변환 등으로 정리된 “정규화된 데이터”만 반환한다.

요구사항 3. 서버에서 내려오는 에러 코드(예: 400, 401, 500 등)를 한 곳에서 처리하는 공통 핸들러를 만든다. 이 핸들러는 에러 코드를 해석해서 사용자에게 보기 쉬운 설명으로 바꿔 주고, 이 에러가 “다시 시도하면 해결될 가능성이 있는지”도 함께 알려 준다.

태스크3. 주소 검색·지오코딩 연동@

요구사항 1. 우편번호/주소 검색 API 호출 서비스와 DTO 변환 로직을 구현해 주소, 우편번호, 도로명 정보를 표준화한다.
요구사항 2. 주소 선택 후 지오코딩 API를 호출해 위도/경도를 받아오고, 사용자가 좌표를 직접 수정하면 역지오코딩으로 주소를 재계산한다.
요구사항 3. 주소·좌표 값을 상태 관리 계층과 공유해 다른 기능(지도, 제출)에 즉시 반영되도록 한다.

태스크4. 지도 상호작용 기능@

요구사항 1. 지도 SDK를 초기화하고 전달받은 좌표로 마커를 렌더링하며, 초기 로딩/실패 상태를 관리한다.
요구사항 2. 마커 드래그 또는 지도 클릭 시 좌표를 재계산해 상태에 반영하고, 반대로 상태 변경 시 지도도 업데이트되도록 양방향 바인딩을 구현한다.
요구사항 3. 확대/축소, 현재 위치 버튼, 좌표 미존재 시 에러 메시지 등 부가 인터랙션을 정의한다.

태스크5. 사진 업로드 파이프라인@

요구사항 1. 다중 파일 선택 후 확장자·용량·최대 장수 검증을 수행해 업로드 큐를 구성한다.
요구사항 2. 업로드 진행률·실패 상태를 추적하고 재시도/취소 API를 노출한다.
요구사항 3. 완료된 파일 메타데이터(URL, 파일명, 용량)를 표준화해 상태에 저장하고 삭제 시 즉시 큐와 상태에서 제거한다.

태스크6. 제출 및 후속 처리 흐름@

요구사항 1. 신규 등록과 수정 모드를 구분해 POST/PUT API를 각각 호출하고 공통 응답 파서를 구현한다.
요구사항 2. 제출 중 중복 요청을 막기 위한 로딩 플래그와 타임아웃·재시도 전략을 적용한다.
요구사항 3. 성공 시 이동 경로(등록 완료 페이지/상세 페이지)와 목록 갱신 트리거를 처리하고, 실패 시 롤백 또는 임시 저장을 수행한다.

태스크7. 로깅·모니터링 구성@
요구사항 1. 주소 검색, 지도 조작, 사진 업로드, 제출 등 주요 이벤트를 추적 로그로 남겨 분석이 가능하도록 한다.
요구사항 2. 클라이언트·서버 오류를 분류해 모니터링 시스템으로 전송하고 알림 임계값을 설정한다.
요구사항 3. 주소·사진 등 개인정보는 마스킹한 뒤 로그에 기록하여 보안 정책을 준수한다.
